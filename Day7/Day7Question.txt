‚ùì What is PRIMARY KEY?
Answer (Interview Ready):
A Primary Key is a column (or combination of columns) that uniquely identifies each record in a table.
It cannot be NULL and must be unique.
ex.
id INT PRIMARY KEY

‚ùì Difference between DELETE & TRUNCATE
| Feature       | DELETE     | TRUNCATE       |
| ------------- | ---------- | -------------- |
| Type          | DML        | DDL            |
| WHERE allowed | ‚úÖ Yes      | ‚ùå No           |
| Rollback      | ‚úÖ Possible | ‚ùå Not possible |
| Speed         | Slower     | Faster         |
| Deletes rows  | One by one | All at once    |
DELETE removes rows conditionally and can be rolled back, whereas TRUNCATE removes all rows permanently and resets the table.

‚ùì DELETE vs DROP
DELETE ‚Üí removes rows
TRUNCATE ‚Üí removes all rows
DROP ‚Üí removes entire table

‚ùì Difference between CHAR vs VARCHAR (TRAP üî•)
| CHAR                  | VARCHAR         |
| --------------------- | --------------- |
| Fixed length          | Variable length |
| Faster for fixed data | Saves space     |
| Wastes memory         | Efficient       |
Use CHAR when data length is fixed (like gender, country code).
Use VARCHAR when length varies (name, email).

‚ùì INT vs BIGINT (TRAP)
| INT            | BIGINT            |
| -------------- | ----------------- |
| 4 bytes        | 8 bytes           |
| Max ~2 billion | Very large values |
Salary may fit in INT today but not tomorrow ‚Üí use BIGINT for safety.

‚ùì What is INDEX?
Index improves SELECT performance by reducing full table scan.
Index slows down INSERT, UPDATE, DELETE.

‚ùì When NOT to use Index?
Small tables
Columns frequently updated
Low cardinality (gender)

‚ùì Can DELETE be rolled back?
‚úÖ Yes (if transaction)

‚ùì Can TRUNCATE be rolled back?
‚ùå No

‚ùì Why TRUNCATE is faster?
It deallocates pages instead of deleting row-by-row.

‚ùì Difference between WHERE and HAVING?
| WHERE                | HAVING              |
| -------------------- | ------------------- |
| Filters rows         | Filters groups      |
| Used before GROUP BY | Used after GROUP BY |
| Cannot use aggregate | Can use aggregate   |
ex.
SELECT dept_id, COUNT(*)
FROM employee
GROUP BY dept_id
HAVING COUNT(*) > 2;
note: WHERE works on rows, HAVING works on aggregated data.

‚ùì INNER JOIN vs LEFT JOIN
| INNER JOIN         | LEFT JOIN           |
| ------------------ | ------------------- |
| Matching rows only | All left table rows |
| Faster             | Slightly slower     |
ex.
SELECT e.name, d.dept_name
FROM employee e
LEFT JOIN department d
ON e.dept_id = d.dept_id;

‚ùì What is normalization?

‚ùì 1NF, 2NF, 3NF (FRESHER LEVEL)

2NF ‚Üí No partial dependency
